# b2p — encrypted file transfer over HTTP upload

**Date:** 2026-07-22
**Status:** Approved design

## Problem

Transfer files between two PCs on different private networks, end-to-end encrypted, like croc — but croc's P2P/relay connections are blocked on these networks. What the networks *do* allow is plain outbound HTTPS ("uploading files to an HTTP site"). No externally hosted infrastructure: only the two PCs run anything.

## Solution overview

The **receiver** hosts a local HTTP server and exposes it via a free Cloudflare quick tunnel (an *outbound* connection from the receiver, so firewalls allow it). The **sender** uploads encrypted chunks to that URL with plain HTTPS requests — indistinguishable from uploading to any website, which is exactly the traffic the restricted network permits.

```
 SENDER PC (restricted network)          RECEIVER PC
 ┌──────────────┐                        ┌──────────────────────────┐
 │  b2p send    │ ── HTTPS POST/PUT ──▶  │ Cloudflare quick tunnel  │
 │  (encrypts)  │    (ciphertext only)   │   ▼ (outbound conn)      │
 └──────────────┘                        │  local HTTP server       │
                                         │  b2p receive (decrypts)  │
                                         └──────────────────────────┘
```

- Rust, single static binary, CLI on both machines.
- End-to-end encrypted: Cloudflare's edge (which terminates TLS) sees only ciphertext.
- `cloudflared` is auto-downloaded on first run (pinned version, checksum-verified); no account needed.
- One transfer per invocation (croc-like); the receiver exits after a completed transfer.

## User flow

1. Receiver: `b2p receive` → generates a secret, starts a localhost HTTP server, opens a quick tunnel, prints a code such as `https://witty-lamp-fox.trycloudflare.com#hV8kPz3q...`. The fragment after `#` is the secret and is never sent over the network.
2. Receiver shares the code out-of-band (chat, email). It is one-time and useless without also being able to intercept the transfer.
3. Sender: `b2p send <code> ./file.zip` (or a folder, several paths, or `--text "snippet"`).
4. Sender POSTs an encrypted manifest. Receiver's terminal prompts: `Incoming: file.zip (1.2 GB) — accept? [y/N]`. The sender's request waits (long-poll) for the answer.
5. On accept, sender encrypts and uploads fixed 4 MiB chunks over HTTPS, a few in parallel (default 4). Both sides show progress bars.
6. Sender posts a final commit containing the whole-content BLAKE3 hash; receiver verifies, unpacks (folders), renames into place, both sides report success.

### CLI

```
b2p receive [--out DIR] [--direct] [--yes] [--overwrite]
b2p send <CODE> <PATH>...
b2p send <CODE> --text "some string"
```

- `--direct`: skip the tunnel; serve on the LAN and print an `http://<lan-ip>:<port>#<secret>` code. Also the backbone of integration tests.
- `--yes`: auto-accept the incoming transfer (scripting).
- Text snippets are printed to the receiver's stdout, not written to a file.

### Code format

`<url>#<secret>` — a real URL (tunnel or direct) plus a fragment carrying the secret: base58 of 16 random bytes (~22 chars, 128-bit entropy). Codes are pasted, not typed; the URL is unguessable anyway, so there is no need for short phrases or PAKE (croc needs PAKE only because its phrases are low-entropy).

## Cryptography

- **Key derivation:** from the 128-bit secret, derive two independent 32-byte keys via `blake3::derive_key` with distinct context strings: an **auth key** and a **data key**.
- **Auth:** every request carries `Authorization: Bearer <base58(auth_key)>`. The server compares in constant time; after 10 failed attempts it shuts down (quick tunnel URLs get scanned by bots).
- **Content encryption:** XChaCha20-Poly1305. Each 4 MiB plaintext chunk is sealed with a deterministic 24-byte nonce: `[domain byte][15 zero bytes][chunk index as u64 LE]`, domains `0x01` = manifest, `0x02` = data chunk, `0x03` = commit; AAD = transfer ID. Nonces never repeat under a key because every transfer has a fresh secret and each (domain, index) pair is used once — so chunks cannot be reordered, replayed, or substituted without AEAD failure.
- **Manifest and commit bodies are encrypted** like chunks; file names never appear in URLs, headers, or plaintext bodies.
- **Whole-transfer integrity:** sender streams a BLAKE3 hash over the plaintext while encrypting and sends it in the commit; receiver verifies before unpacking or renaming anything.

### Threat model

- Cloudflare's edge terminates TLS and can read HTTP traffic. It sees: ciphertext bodies, chunk indices in URLs, request sizes/timing, and the auth token. The auth token cannot decrypt anything (derivation is one-way); the worst a malicious edge can do is inject garbage, which AEAD tags reject.
- Out of scope: hiding transfer metadata (size, timing, chunk count), and attackers who obtain the code itself — the code is the root secret and must be shared over a channel the user trusts.

## Wire protocol

All endpoints require the Authorization header. Bodies that carry names or content are ciphertext; chunk indices and progress bitmaps are not separately encrypted (indices are already visible in URLs to the TLS-terminating edge — no additional leak).

| Endpoint | Purpose |
|---|---|
| `POST /v1/manifest` | Encrypted manifest: version, transfer ID, mode (files/text), entries (path, size), total size, chunk size; inline content for text mode. Long-polls until the receiver accepts/denies (sender times out after 120 s with a clear message). Response includes the bitmap of chunks the receiver already has (resume). |
| `PUT /v1/chunk/{index}` | One encrypted chunk. Server decrypts, writes at `index × 4 MiB`, records the index. |
| `GET /v1/status` | Current received-chunk bitmap (sender re-sync after network drops). |
| `POST /v1/commit` | Encrypted whole-content BLAKE3 hash. Server verifies completeness + hash, finalizes, responds success/failure. |

A protocol `version` field in the manifest lets either side reject a mismatched peer cleanly.

## Resume

- Receiver writes decrypted chunks into `<out>/<name>.b2p-partial/` (data file + `state.json` with the chunk bitmap), synced as chunks arrive.
- **Transfer ID** = BLAKE3 fingerprint of the source metadata (relative paths, sizes, mtimes). Cheap (no upfront hashing of content), stable across retries.
- On a new manifest, the receiver matches transfer ID against partial state and replies with the chunks it already has; the sender uploads only the gaps.
- Symmetric recovery: sender died → re-run the same command (receiver still up, same code). Receiver died → re-run `b2p receive` (new code — quick tunnel URLs are ephemeral), sender re-runs with the new code, fingerprint match resumes the same partial data.
- Chunk independence (bitmap, not sequence) is what allows both parallel uploads and out-of-order arrival.
- Finalize: verify bitmap complete + BLAKE3 match → single file renamed into place; folder/multi-file transfers unpacked from the tar stream, then partial dir removed. Folder transfers therefore briefly need ~2× the transfer size free on the receiver.

## Data handling

- Folders / multiple paths are wrapped in a single tar stream sender-side and unpacked receiver-side after verification; the protocol only ever carries one byte stream.
- Single file: raw stream, no tar (flagged in manifest).
- Text snippet: carried inline in the manifest, printed to stdout.
- Overwrite protection: existing destination file prompts unless `--overwrite`.

## Module structure

```
b2p/
├─ src/
│  ├─ main.rs        CLI entry (clap): `receive`, `send`
│  ├─ code.rs        generate/parse the code string (url + secret)
│  ├─ crypto.rs      key derivation, chunk seal/open, auth token — pure functions
│  ├─ archive.rs     tar-stream folders/multi-file; passthrough single file; text snippets
│  ├─ tunnel.rs      TunnelProvider trait + cloudflared impl (download, spawn, parse URL, cleanup)
│  ├─ server.rs      receiver HTTP server (axum): manifest/chunk/status/commit routes, auth middleware
│  ├─ store.rs       receiver transfer state: partial file, chunk bitmap, finalize
│  ├─ send.rs        sender client (reqwest): handshake, parallel uploads, resume, commit
│  └─ progress.rs    progress bars (indicatif)
```

Dependency direction: `server` and `send` sit on `crypto` + `archive`; `tunnel` is behind a trait so `--direct` mode and future providers plug in without touching anything else; `crypto`, `code`, `store`, `archive` have no networking and are fully unit-testable.

Main crates: `tokio`, `axum`, `reqwest` (rustls), `chacha20poly1305`, `blake3`, `clap`, `tar`, `indicatif`, `serde`.

### cloudflared management

Pinned release version with per-platform SHA-256 checksums baked into the source; downloaded from Cloudflare's official GitHub releases into the platform data dir (e.g. `~/.local/share/b2p/`) on first use; spawned as a child process (quick tunnel mode), URL parsed from its output, killed on exit/drop.

## Error handling

- **Tunnel won't start** (some networks block cloudflared's own connection): detect, explain plainly, suggest `--direct` or running the receiver on a friendlier network. By design this is only ever the receiver's problem — the sender needs nothing but plain HTTPS.
- **Flaky tunnel mid-transfer** (free-tunnel 502s): per-chunk retry with exponential backoff; on giving up, print the exact command to resume.
- **Ctrl-C anywhere:** both sides always leave resumable state; resume is the universal recovery path.
- **Receiver safety:** overwrite prompt; disk-full surfaced clearly with partial state kept; 10 bad auth attempts → server exits.

## Testing

- **Unit:** crypto round-trip + tamper rejection (bit-flip in ciphertext, swapped chunk indices), code parse/format, bitmap/resume logic in `store`, tar round-trip in `archive`.
- **Integration (CI, no network):** full send↔receive over localhost via `--direct` — happy path (file, folder, text), kill-mid-transfer + resume, wrong-token rejection, corrupted-chunk rejection.
- **Manual smoke script:** real tunnel end-to-end (depends on Cloudflare's live service; not in CI).

## Out of scope (v1)

- Browser-based receiving (would need in-browser decryption).
- Sender-hosts or negotiated-direction modes (the tunnel trait leaves the door open).
- Multiple transfers per server invocation; transfer queues.
- Metadata privacy (sizes/timing) beyond content encryption.
