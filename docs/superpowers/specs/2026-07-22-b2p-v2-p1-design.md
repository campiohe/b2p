# b2p v2 — Phase P1 Design

*Real peer-to-peer transfer that works on the proving-ground network without any tunnel: PAKE codes + ntfy rendezvous + WebRTC(STUN) data channel.*

Status: **Approved for planning** · Builds on P0 (merged: OS trust store, `b2p doctor`, layered errors) · Implements spec `b2p-v2-spec.md` §§3–5, phase P1.

---

## 0. Goal and boundary

P1 delivers the spec's headline: `b2p receive` and `b2p send` complete a transfer over a WebRTC data channel, meeting on a free no-account rendezvous (`ntfy.sh`), with a short human code turned into a strong key by a PAKE. The v1 Cloudflare-tunnel path from P0 stays fully intact and is selected only when the code is a v1 `https://…` URL or the user passes `--tunnel`.

**In P1:** human + URL code formats, SPAKE2 key agreement, a pluggable `Rendezvous` trait with an ntfy implementation, a `webrtc`-crate transport using STUN only, and reuse of the existing manifest/chunk/commit protocol + chunk `Store` over the data channel.

**Not in P1 (later phases, unchanged from the master spec):** TURN fallback and TURN-over-443 (P2); self-hosted relay transport (P2); the full parallel negotiation budget engine (P2); mDNS LAN auto-direct, store-and-forward, and cross-transport resume (P3). P1's "negotiation" is trivial: the code form plus `--tunnel` decides the path.

**Non-negotiable constraints (inherited from P0):**
- `cargo test` stays fully offline. WebRTC, ntfy, and STUN are exercised only through pure functions, in-process loopback peers, and localhost fakes — never the public internet.
- CI gate stays green: `cargo fmt --check`, `cargo clippy -- -D warnings`, `cargo test`.
- rustls only, no OpenSSL; all outbound HTTPS goes through `crate::http::client`/`builder` (the OS-trust factory), honoring `--cafile`.
- End-to-end encryption never depends on the transport: payload is AEAD-sealed under the PAKE session key regardless of DTLS.

## 1. The code, the topic, and the secret

A code encodes exactly two things (spec §3.1): a **rendezvous topic** (which ntfy mailbox to meet on) and a **PAKE secret** (the low-entropy password SPAKE2 consumes). Three spellings parse; two are new.

### 1.1 Human code — the new default

`<channel>-<word>-<word>`, e.g. `7-tunnel-mosaic`.

- `channel`: an integer 0–255.
- `word`, `word`: two entries from a fixed 256-word list (`src/words.rs`, ~4–8 char lowercase ASCII, no confusable pairs). Wordlist is frozen at first release; changing it is a breaking code change.
- These three symbols decode to **3 entropy bytes** `E = [channel, word_index_1, word_index_2]` (24 bits). This is deliberately low-entropy — that is what the PAKE is for.
- **Topic** = `base58( HKDF("b2p-v2-rendezvous-topic", E)[..16] )` — a 16-byte, effectively unguessable-by-accident topic id. (Enumerable by a determined attacker; §5 explains why PAKE + single-use + attempt-limit contains that.)
- **PAKE password** = `E` (all 3 bytes).

The receiver generates `E` with a CSPRNG, prints the human code, and derives the topic. The sender parses the same human code, derives the same topic, and both run SPAKE2 with `E`.

### 1.2 URL code — power-user / explicit form

`b2p://<topic-b58>#<secret-b58>`

- `topic-b58`: the 16-byte rendezvous topic, explicit (may be published; carries no secret).
- `secret-b58`: a 16-byte high-entropy PAKE secret, in the URL fragment — **never** sent to any server (same fragment discipline as v1).
- Here topic and secret are independent random values (not derived from each other). SPAKE2 runs with the full 16-byte secret.

This form is offered for scripts and QR flows where a full-entropy secret is preferable and typing isn't required.

### 1.3 v1 compatibility

`https://<host>.trycloudflare.com#<secret>` still parses (P0's `Code`) and routes to the **tunnel** transport. `b2p send` inspects the code:
- `https://` or `http://` → tunnel path (P0 code, unchanged).
- `b2p://` or human form → PAKE + rendezvous + WebRTC path (P1).

`--tunnel` on `receive` forces the v1 path and prints an `https://` code; without it, `receive` prints a human code and runs the P1 stack.

### 1.4 Module shape

Additive-first, so P1a disturbs no existing behavior:

- `src/words.rs` — the frozen `WORDS: [&str; 256]` array + `index_of(word) -> Option<u8>`.
- `src/rvcode.rs` (new) — `RendezvousCode { topic: String, secret: PakeSecret }` with human (`<channel>-<word>-<word>`) and URL (`b2p://<topic>#<secret>`) parse/render, and human ↔ `E` encode/decode helpers. The existing P0 tunnel `Code` (`src/code.rs`) is left untouched in P1a/P1b.
- **P1c** adds the top-level dispatch — a small `parse_any(&str)` that returns either the tunnel `Code` (`https://…`) or a `RendezvousCode` (`b2p://` / human) and drives transport selection in `main.rs`/`send.rs`. Unifying the two into one enum is deferred to that integration step, where the call sites change anyway.

## 2. PAKE handshake

`src/pake.rs`, built on the `spake2` crate (pure Rust; the implementation magic-wormhole-rs uses).

- **Symmetric SPAKE2** (balanced; both sides prove knowledge of `E`/secret). Identity string binds the transcript to `"b2p-v2"` and the topic so a message replayed on another topic fails confirmation.
- Flow over the rendezvous channel:
  1. Each side computes its SPAKE2 message from the password and publishes it (`pake/A`, `pake/B` framing so a side ignores its own).
  2. On receiving the peer's message, each derives the shared 256-bit key material.
  3. **Key confirmation:** each side sends `HMAC(key, "confirm"|role)`; both verify the peer's before any payload flows. A wrong code → confirmation mismatch → abort with a clear "code did not match — check it and retry" error. This is what makes an active MITM on ntfy detectable (spec §9).
- Output: `SessionKey([u8; 32])`. From it, HKDF derives the AEAD stream key (`Domain::Stream`) and any SDP-encryption key — independent subkeys, same pattern as `crypto.rs`'s `derive_key` domains.
- **Single-use:** the receiver accepts exactly one successful confirmation per code; subsequent handshake attempts on that topic are refused. A configurable timeout (default 10 min) expires an unused code.

**Interface (consumed by P1b/P1c):**
```
pub struct SessionKey(pub [u8; 32]);
pub enum Role { Receiver, Sender }
// Drives the full handshake over any Rendezvous, returns the confirmed key.
pub async fn handshake(rv: &dyn Rendezvous, topic: &str, password: &[u8], role: Role)
    -> anyhow::Result<SessionKey>;
```

## 3. Rendezvous layer

`src/rendezvous/mod.rs` (trait) + `src/rendezvous/ntfy.rs` (default impl).

```rust
#[async_trait]
pub trait Rendezvous: Send + Sync {
    /// Publish one frame to the topic. Frames are opaque bytes (already
    /// encrypted or public PAKE values).
    async fn publish(&self, topic: &str, frame: &[u8]) -> anyhow::Result<()>;
    /// Subscribe to frames on the topic. Yields each frame body.
    async fn subscribe(&self, topic: &str) -> anyhow::Result<BoxStream<'static, Vec<u8>>>;
    async fn close(&self) -> anyhow::Result<()>;
}
```

- **ntfy framing:** frames are POSTed to `https://ntfy.sh/<topic>` as the request body, base64-wrapped (ntfy messages are UTF-8 text) with a 1-byte frame-type tag (`pake-a`, `pake-b`, `confirm`, `sdp-offer`, `sdp-answer`, `ice`). Subscribe reads the JSON stream (`/<topic>/json`) and decodes bodies. A subscriber ignores frames of types it didn't expect and its own frames (a self-published nonce/sender-id tag disambiguates the two peers).
- Built on `crate::http::client(&TlsOpts)` — inherits OS-trust + `--cafile`, so it works through the inspecting proxy exactly as `b2p doctor` already verifies.
- **Pluggable:** `--rendezvous <url>` swaps the ntfy base URL (self-hosted ntfy). The trait boundary keeps alternative providers a later drop-in without touching PAKE or transport.
- Only PAKE public values and E2E-encrypted SDP/ICE transit the rendezvous — never the code, key, filenames, or content (spec §9 table).

## 4. WebRTC transport

`src/transport/mod.rs` (trait + selection) + `src/transport/webrtc.rs` (impl) + `src/stream.rs` (framed AEAD payload over a duplex channel).

### 4.1 Transport trait (minimal, P2-ready)

```rust
pub enum TransportKind { WebRtc, Tunnel }

#[async_trait]
pub trait Transport {
    /// After signaling completes, a bidirectional reliable byte channel.
    async fn open(self: Box<Self>) -> anyhow::Result<Duplex>;
    fn kind(&self) -> TransportKind;
}
```
`Duplex` is a reliable, ordered `AsyncRead + AsyncWrite` (the SCTP data channel wrapped so `stream.rs` doesn't know it's WebRTC). The tunnel path is **not** retrofitted to this trait in P1 — it keeps its own `send.rs`/`server.rs` code, selected by code form. The trait exists so P2's relay implements it and the negotiation engine has something to negotiate.

### 4.2 WebRTC establishment (`webrtc` crate)

- One reliable, ordered `RTCDataChannel` ("b2p").
- ICE gathers **host + server-reflexive (STUN)** candidates only — public no-account STUN servers (`stun.l.google.com:19302`, `stun.cloudflare.com:3478`), the ones `b2p doctor` already probes. **No TURN in P1.**
- SDP offer/answer and trickled ICE candidates are exchanged as frames over the `Rendezvous`, each **encrypted** under an HKDF subkey of the session key (so ntfy sees only ciphertext, per spec §9). Receiver creates the offer; sender answers (or vice-versa — fixed by `Role`).
- **Budget:** ICE gather + connectivity checks get ~8 s (spec §5). If no candidate pair reaches "connected", abort and auto-run `b2p doctor` — never hang. Diagnosis names the layer ("UDP/STUN worked but no candidate pair formed — likely symmetric NAT; P1 has no TURN yet, try `--tunnel` or the same LAN").

### 4.3 Payload framing (`src/stream.rs`)

Reuses `crypto.rs`, not libsodium. Over the `Duplex`:

- **Frame:** `u32 length prefix` + XChaCha20-Poly1305 ciphertext. `Domain::Stream` (new domain byte), per-frame monotonic index as the nonce counter, session-derived key. 64 KiB plaintext frames.
- **First frame:** the `Manifest` (existing struct from `protocol.rs`) — filenames, sizes, kind. Receiver runs the same accept prompt / auto-accept logic as P0.
- **Middle frames:** payload bytes (the raw file for `Kind::File`, or the tar spool for `Kind::Tar`) sealed 64 KiB at a time, streamed straight into the existing `Store` so integrity + finalize (blake3 commit) is identical to P0.
- **Final frame:** a marker (`is_final` AAD bit or a zero-length terminal frame carrying the `Commit`) so truncation is detected — receiver verifies the blake3 before finalizing, exactly as `server.rs::finalize` does today.
- Text transfers (`Kind::Text`) complete in the manifest frame, as in P0.

Reusing `Store` means the on-disk staging, blake3 verification, and tar-unpack paths are shared between tunnel and WebRTC transfers — one finalize implementation, one integrity guarantee.

## 5. Security model (P1 specifics)

Holds the spec §9 model. Additions/confirmations for P1:

- **Low-entropy human code is safe** against an untrusted ntfy because SPAKE2 gives no offline-dictionary foothold (the operator sees only SPAKE2 public values), and the receiver enforces **single-use** (one confirmed handshake per code) + **attempt-limit** (a small number of failed confirmations retires the topic) + **timeout**. An online guesser gets one shot per code before it dies.
- **Topic enumeration** (human topics derive from 24-bit `E`): an attacker can sit on a topic, but learns nothing without the code and can only force one failed PAKE attempt before single-use/attempt-limit closes it. Interference is a DoS at worst, never a confidentiality break.
- **Active MITM on signaling** is caught at key confirmation (§2) before any SDP or payload flows; the transfer aborts.
- **ntfy / STUN / any observer** sees ciphertext SDP, ciphertext payload, timing, and volume — never the key, filenames, or content. STUN servers see only the reflexive-candidate request, never payload.
- The permissive TLS-capture verifier from P0 (`tlsprobe.rs`) remains diagnostic-only; the P1 data and rendezvous paths use real OS-trust verification.

## 6. CLI & UX changes

```
b2p receive [--out DIR] [--tunnel] [--rendezvous URL] [--yes] [--overwrite] [--cafile FILE]
    default → prints a human code (e.g. 7-tunnel-mosaic) and a b2p:// URL; runs PAKE+ntfy+WebRTC
    --tunnel → P0 behavior: cloudflared tunnel + https:// code
    → "Waiting for sender…" with live status: rendezvous connected → PAKE ok → gathering ICE → via WebRTC (STUN)

b2p send <code> [PATHS…] [--text TEXT] [--rendezvous URL] [--cafile FILE]
    code form selects the transport automatically (https:// → tunnel, b2p:// or human → WebRTC)
    → shows the chosen path ("via WebRTC (STUN)") and streams with the existing progress bar
```

- **Transparency (spec §8):** always print which transport won and how (`via WebRTC (STUN)`, `via tunnel`).
- **Failure:** WebRTC establishment failure runs `b2p doctor` automatically and prints a layered, specific message — no generic error.

## 7. Execution plan (three sub-plans)

Each sub-plan lands merged, green, and reviewed before the next starts — same subagent-driven workflow as P0.

| Sub-plan | Scope | Independently testable deliverable |
| --- | --- | --- |
| **P1a — codes + PAKE** | `words.rs`, `code.rs` (enum + human/URL parse/render), `pake.rs` (SPAKE2 + confirmation). No network. | Two in-process peers sharing an in-memory channel complete a handshake to the same `SessionKey`; wrong code → confirmation failure. Code round-trips for all three spellings. |
| **P1b — rendezvous + ntfy** | `rendezvous/mod.rs` trait, `rendezvous/ntfy.rs`, frame encode/decode. PAKE from P1a runs over the trait. | Against a **localhost** fake ntfy (axum) server: two peers publish/subscribe and complete the P1a handshake end-to-end over the trait. Frame codec unit-tested. |
| **P1c — WebRTC + integration** | `transport/mod.rs`, `transport/webrtc.rs`, `stream.rs`; wire `main.rs`/`send.rs`/`receive` for code-form transport selection, default-to-P1, `--tunnel` escape, doctor-on-failure. | Two in-process peers form a `webrtc`-crate data channel over an **in-memory/loopback** signaling shim and transfer a multi-chunk file + a folder + text, verified byte-identical via the reused `Store`. Selection logic unit-tested (code form → transport). Live end-to-end over real ntfy/STUN is a manual smoke script (`scripts/smoke-webrtc.sh`), not part of `cargo test`. |

Dependencies: `spake2`, `hkdf` (or blake3's KDF, already vendored), `webrtc`, `async-trait`, `base64`. All additions rustls-compatible; `webrtc` uses its own bundled crypto for DTLS/SRTP but carries no OpenSSL.

## 8. Risks & mitigations

- **`webrtc` crate dependency weight / API churn.** Mitigation: confine all of it to `transport/webrtc.rs` behind the `Transport` trait; the rest of the code sees only `Duplex`. If the crate proves unworkable during P1c, the trait lets `str0m` substitute without touching P1a/P1b.
- **Offline-test fidelity for WebRTC.** Loopback data-channel tests exercise the real crate over localhost UDP, so ICE/DTLS/SCTP run for real without the public internet; STUN reflexive gathering is the one piece only the manual smoke script covers.
- **Human-code entropy vs. usability.** 24 bits is wormhole-class and adequate given single-use + attempt-limit; if collision/DoS proves annoying in practice, the channel range widens without a format change.
- **Two crypto call-sites (PAKE key → stream key, PAKE key → SDP key).** Both derive via one HKDF domain scheme in `pake.rs`; no key is reused across domains.
